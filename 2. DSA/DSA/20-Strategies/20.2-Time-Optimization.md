## Complexity Targets

Use constraint size to reverse-engineer the required complexity before writing a single line of code.

| Input Size (n) | Required Complexity | Typical Approach |
|---------------|--------------------|--------------------|
| n ≤ 15 | O(2ⁿ) or O(n!) | Bitmask DP, backtracking |
| n ≤ 100 | O(n³) | Floyd-Warshall, 3D DP |
| n ≤ 1 000 | O(n²) | 2D DP, nested loops |
| n ≤ 10 000 | O(n² ) with small constant | DP with pruning |
| n ≤ 100 000 | O(n log n) | Sort, segment tree, heap |
| n ≤ 1 000 000 | O(n) | Two pointers, sliding window, prefix sum |
| n ≤ 10⁹ | O(log n) | Binary search, fast exponentiation |

## Pruning

### What It Is
Cutting off branches of a search or recursion tree that cannot possibly lead to a better answer.

### Technique 1: Early Return

```java
// Without pruning: visits every node even when answer is obvious
boolean dfs(int[] nums, int idx, int target) {
    if (idx == nums.length) return target == 0;
    return dfs(nums, idx + 1, target - nums[idx]) || dfs(nums, idx + 1, target);
}

// With pruning: stop when remaining sum cannot match target
boolean dfs(int[] nums, int idx, int target, int remainingSum) {
    if (target == 0) return true;
    if (target < 0 || idx == nums.length) return false;
    if (remainingSum < target) return false;       // ← pruning cut
    return dfs(nums, idx + 1, target - nums[idx], remainingSum - nums[idx])
        || dfs(nums, idx + 1, target, remainingSum - nums[idx]);
}
```

### Technique 2: Sorted Input + Break Early

```java
// Combination Sum II — skip if current element already exceeds target
Arrays.sort(candidates);
void backtrack(int[] candidates, int start, int target, List<Integer> path) {
    if (target == 0) { result.add(new ArrayList<>(path)); return; }
    for (int i = start; i < candidates.length; i++) {
        if (candidates[i] > target) break;     // sorted → all remaining are larger
        if (i > start && candidates[i] == candidates[i - 1]) continue; // skip duplicates
        path.add(candidates[i]);
        backtrack(candidates, i + 1, target - candidates[i], path);
        path.remove(path.size() - 1);
    }
}
```

### Technique 3: Bound Estimation (Branch and Bound)

```java
// TSP-style: if current partial path + lower bound >= best so far, prune
if (currentCost + lowerBound(remaining) >= bestSoFar) return;
```

## Memoization

### Top-Down with HashMap

```java
Map<String, Long> memo = new HashMap<>();

long solve(int[] nums, int idx, int target) {
    String key = idx + "," + target;
    if (memo.containsKey(key)) return memo.get(key);
    // base case
    if (idx == nums.length) return target == 0 ? 1 : 0;
    long result = solve(nums, idx + 1, target - nums[idx])
                + solve(nums, idx + 1, target);
    memo.put(key, result);
    return result;
}
// String key is slow — encode as long when possible
```

### Faster Key Encoding

```java
// Encode two 32-bit ints into one long
long key = ((long) idx << 32) | (target & 0xFFFFFFFFL);
```

### When Memoization Beats Tabulation

- State space is **sparse** (only a fraction of states are reachable).
- Recursion naturally skips unreachable states; tabulation fills them all.
- Example: word break, regex matching with many impossible prefixes.

### Converting Top-Down to Bottom-Up (Tabulation)

1. Identify all states and their dimensions.
2. Determine iteration order so that `dp[i]` uses only already-computed states.
3. Replace recursive calls with array lookups.

```java
// Top-down: fibonacci
Map<Integer, Long> memo = new HashMap<>();
long fib(int n) {
    if (n <= 1) return n;
    if (memo.containsKey(n)) return memo.get(n);
    long result = fib(n - 1) + fib(n - 2);
    memo.put(n, result);
    return result;
}

// Bottom-up equivalent: O(n) time, O(n) space (reducible to O(1))
long[] dp = new long[n + 1];
dp[0] = 0; dp[1] = 1;
for (int i = 2; i <= n; i++) dp[i] = dp[i - 1] + dp[i - 2];
```

## Choosing Better Data Structures

The right data structure can turn an O(n²) solution into O(n log n) or O(n).

| Operation Needed | Naive | Better |
|-----------------|-------|--------|
| Range min/max query (static) | O(n) scan | O(1) Sparse Table (O(n log n) build) |
| Range sum (dynamic point updates) | O(n) | O(log n) Fenwick Tree / Segment Tree |
| Order statistics (kth smallest) | O(n log n) sort | O(log n) Order-Statistic Tree |
| Median of stream | O(n log n) sort | O(log n) two-heap trick |
| Nearest smaller element | O(n²) brute | O(n) Monotonic Stack |
| Frequent element in range | O(n) scan | O(√n) Mo's algorithm |
| Connectivity queries | O(n) BFS/DFS | O(α(n)) Union-Find |

### Two-Heap Trick (Median of Stream)

```java
PriorityQueue<Integer> lo = new PriorityQueue<>(Collections.reverseOrder()); // max-heap
PriorityQueue<Integer> hi = new PriorityQueue<>();                            // min-heap

void addNum(int num) {
    lo.offer(num);
    hi.offer(lo.poll());          // balance: push lo's max into hi
    if (lo.size() < hi.size()) lo.offer(hi.poll()); // keep lo >= hi in size
}

double findMedian() {
    return lo.size() > hi.size() ? lo.peek() : (lo.peek() + hi.peek()) / 2.0;
}
// Time: O(log n) per add | Space: O(n)
```

## Amortized Tricks

### The Monotonic Stack Insight
Each element is pushed and popped at most once → O(n) total even though the inner `while` loop looks like O(n).

```java
// Total pushes = n, total pops ≤ n → O(n) amortized
for (int i = 0; i < n; i++) {
    while (!stack.isEmpty() && nums[stack.peek()] < nums[i]) stack.pop();
    stack.push(i);
}
```

### HashMap Amortized Resize
`HashMap.put` is O(1) amortized (O(n) on rehash, but rarely). Pre-size to avoid rehashing:

```java
Map<Integer, Integer> map = new HashMap<>(initialCapacity * 4 / 3 + 1);
// or simply new HashMap<>(n * 2)
```

### Union-Find α(n) Amortized
With path compression + union by rank, `find` and `union` cost O(α(n)) amortized where α is the inverse Ackermann function — effectively constant for all practical n.

## Two-Pass and Pre-Processing

### Left-Right Pass

Pre-compute prefix and suffix arrays to answer each query in O(1):

```java
// Product of array except self (LC 238) — no division
int[] left = new int[n], right = new int[n], result = new int[n];
left[0] = 1;
for (int i = 1; i < n; i++) left[i] = left[i - 1] * nums[i - 1];
right[n - 1] = 1;
for (int i = n - 2; i >= 0; i--) right[i] = right[i + 1] * nums[i + 1];
for (int i = 0; i < n; i++) result[i] = left[i] * right[i];
// Time: O(n) | Space: O(n) (reducible to O(1) extra)
```

### Precomputed Frequency / Cumulative Structures

```java
// Count of each character in O(1) for any substring
// Build prefix counts for all 26 characters
int[][] prefix = new int[n + 1][26];
for (int i = 0; i < n; i++) {
    prefix[i + 1] = prefix[i].clone();
    prefix[i + 1][s.charAt(i) - 'a']++;
}
// count of char c in s[l..r]: prefix[r+1][c] - prefix[l][c]
```

## Lazy Evaluation

Defer expensive computation until it is actually needed.

### Lazy Propagation in Segment Trees

```java
// Range update without lazy: O(n) per update
// With lazy: O(log n) per update — push down only when querying children

void pushDown(int node) {
    if (lazy[node] != 0) {
        tree[2*node]   += lazy[node] * (mid - left + 1);
        tree[2*node+1] += lazy[node] * (right - mid);
        lazy[2*node]   += lazy[node];
        lazy[2*node+1] += lazy[node];
        lazy[node] = 0;
    }
}
```

### Memoize on First Access

```java
Integer cachedResult = null;
int expensiveCompute() {
    if (cachedResult == null) cachedResult = /* heavy computation */;
    return cachedResult;
}
```

## Bit-Level Speedups

```java
// Check if n is a power of 2 → O(1) vs O(log n) division loop
boolean isPow2 = (n > 0) && (n & (n - 1)) == 0;

// Fast modulo for power-of-2 modulus
int mod = 1 << 20;          // 2^20
int remainder = x & (mod - 1);  // same as x % mod but O(1)

// Count set bits (Integer.bitCount is intrinsified to POPCNT on modern JVMs)
int bits = Integer.bitCount(n);  // O(1) hardware instruction

// Lowest set bit (used in Fenwick Tree index advance)
int lsb = n & (-n);

// Enumerate all subsets of a bitmask
for (int sub = mask; sub > 0; sub = (sub - 1) & mask) {
    // process sub
}
// Total iterations across all masks = 3^n (not 4^n)
```

## Optimization Checklist

Work through this before declaring your solution final:

1. **Complexity match** — does your O align with the constraint table above?
2. **Unnecessary recomputation** — add memoization or precompute prefix arrays.
3. **Wrong data structure** — HashMap O(1) vs TreeMap O(log n) vs array O(1).
4. **Sorting enables better algorithm** — sort first, then two-pointer or binary search.
5. **Pruning possible** — can you cut branches early in recursion / backtracking?
6. **Inner loop doing redundant work** — precompute before the loop.
7. **Amortized cost hidden** — monotonic stack, union-find, hash table are cheap amortized.
8. **Constant factor** — prefer `int[]` over `Integer[]`, `ArrayDeque` over `Stack`, `StringBuilder` over string concatenation.

---

## Key Takeaways

- Match n to the required complexity *before* coding; it tells you which algorithm class to use.
- Pruning is free — add it to backtracking whenever you can bound remaining possibilities.
- Memoize when subproblems overlap; choose hash key encoding carefully for speed.
- Switching data structures (stack→monotonic stack, list→heap, array→segment tree) is often the entire optimization.
- Amortized analysis explains why monotonic stack, union-find, and hash tables are effectively O(1).
- Two-pass (prefix + suffix) patterns unlock O(1) per query after O(n) pre-processing.
- Bit tricks give constant-time operations that would otherwise require loops.
